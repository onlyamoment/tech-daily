oMLX

**为 Mac 优化的 LLM 推理引擎**

连续批处理、分层 KV 缓存，直接从菜单栏管理。

---

简介

oMLX 是一个面向 Apple Silicon 的本地大模型推理服务。它解决的核心问题是：现有 LLM 服务端往往在易用性和可控性之间二选一。oMLX 的设计目标是把日常模型常驻内存、重模型按需自动切换、上下文长度可配置——并且全部通过菜单栏完成。

它的 KV 缓存在内存热层和 SSD 冷层之间持久化。即使对话中途切换上下文，过往内容依然缓存并可复用，这让本地 LLM 在配合 Claude Code 等工具做真实开发时变得更实用。

---

安装方式

**macOS App**：下载 `.dmg` 拖入应用程序即可，支持应用内自动更新。

**Homebrew**：

```bash

brew tap jundot/omlx https://github.com/jundot/omlx

brew install omlx

```

**源码安装**：

```bash

git clone https://github.com/jundot/omlx.git

cd omlx

pip install -e .

```

要求：macOS 15.0+，Python 3.10+，Apple Silicon（M1/M2/M3/M4）。

---

快速开始

**App 方式**：启动后按欢迎向导完成模型目录、服务启动和首次下载即可。

**CLI 方式**：

```bash

omlx serve --model-dir ~/models

```

服务自动发现 LLM、VLM、嵌入模型和重排序模型。任何兼容 OpenAI API 的客户端都可以连接到 `http://localhost:8000/v1`。内置聊天 UI 在 `http://localhost:8000/admin/chat`。

---

主要功能管理面板

提供实时监控、模型管理、聊天、基准测试和单模型设置。支持中英日韩俄五种语言，所有 CDN 依赖均已打包，可完全离线使用。

视觉语言模型（VLM）

与文本模型共用同一套连续批处理和分层 KV 缓存。支持多图聊天、base64/URL/文件输入、带视觉上下文的工具调用。OCR 模型（DeepSeek-OCR、DOTS-OCR、GLM-OCR）自动检测并加载优化提示词。

分层 KV 缓存（热层 + 冷层）

借鉴 vLLM 的块式 KV 缓存管理，支持前缀共享和写时复制。两层架构：

  • **热层（内存）**：高频访问块常驻内存，读取速度快。
  • **冷层（SSD）**：热层满后，块以 safetensors 格式卸载到 SSD。下次请求命中相同前缀时，直接从磁盘恢复，无需重新计算——即使服务器重启后也能恢复。
连续批处理

通过 mlx-lm 的 BatchGenerator 处理并发请求，最大并发数可通过 CLI 或管理面板配置。

Claude Code 优化

支持上下文缩放，让小上下文模型配合 Claude Code 使用时，自动压缩触发时机更合理；SSE 保活机制防止长预填充阶段的读取超时。

多模型服务

同一服务端内可加载 LLM、VLM、嵌入模型和重排序模型。管理机制包括：

  • **LRU 淘汰**：内存不足时自动卸载最少使用的模型。
  • **手动加载/卸载**：管理面板内交互式操作。
  • **模型固定**：把常用模型钉在内存中，避免被卸载。
  • **单模型 TTL**：设置空闲超时时间，到期自动卸载。
  • **进程内存限制**：总内存上限默认为系统内存减 8GB，防止系统级 OOM。
单模型设置

可在管理面板直接配置采样参数、聊天模板参数、TTL、模型别名、模型类型覆盖等，改动即时生效，无需重启服务。

内置聊天

支持对话历史、模型切换、深色模式、推理模型输出、VLM/OCR 图片上传。

模型下载器

直接在管理面板搜索并下载 HuggingFace 上的 MLX 模型，浏览模型卡、查看文件大小、一键下载。

一键集成

管理面板内一键配置 OpenClaw、OpenCode、Codex 和 Pi，无需手动编辑配置文件。

性能基准测试

一键测量预填充（PP）和文本生成（TG）的每秒 token 数，并做部分前缀缓存命中测试，给出贴近真实场景的性能数据。

macOS 菜单栏应用

原生 PyObjC 实现的菜单栏应用（非 Electron），可启动、停止和监控服务，无需打开终端。包含持久化服务统计、崩溃自动重启和应用内自动更新。

API 兼容性

兼容 OpenAI 和 Anthropic API 的即插即用替代方案。支持流式用量统计、Anthropic 自适应 thinking、视觉输入（base64、URL）。主要端点包括聊天补全、文本补全、Anthropic Messages、嵌入、重排序和模型列表。

工具调用与结构化输出

支持 mlx-lm 提供的所有函数调用格式、JSON Schema 验证和 MCP 工具集成。自动检测的模型家族包括 Llama、Qwen、DeepSeek、Gemma、GLM、MiniMax、Mistral、Kimi K2、Longcat 等。

---

支持的模型

支持 mlx-lm 涵盖的所有 LLM，mlx-vlm 涵盖的 VLM，以及专用 OCR、嵌入和重排序模型。模型按目录自动检测类型，也支持两级目录结构（如 `mlx-community/model-name/`）。

| 类型 | 代表模型 |

|------|---------|

| LLM | mlx-lm 支持的任意模型 |

| VLM | Qwen3.5 系列、GLM-4V、Pixtral 等 |

| OCR | DeepSeek-OCR、DOTS-OCR、GLM-OCR |

| 嵌入 | BERT、BGE-M3、ModernBERT |

| 重排序 | ModernBERT、XLM-RoBERTa |

---

CLI 配置示例

```bash

内存限制

omlx serve --model-dir ~/models --max-model-memory 32GB

进程级内存上限（默认：系统内存 - 8GB）

omlx serve --model-dir ~/models --max-process-memory 80%

启用 SSD 缓存

omlx serve --model-dir ~/models --paged-ssd-cache-dir ~/.omlx/cache

热层大小

omlx serve --model-dir ~/models --hot-cache-max-size 20%

最大并发请求（默认：8）

omlx serve --model-dir ~/models --max-concurrent-requests 16

MCP 工具

omlx serve --model-dir ~/models --mcp-config mcp.json

HuggingFace 镜像

omlx serve --model-dir ~/models --hf-endpoint https://hf-mirror.com

API Key 认证

omlx serve --model-dir ~/models --api-key your-secret-key

```

所有设置也可在 `/admin` 管理面板中配置，持久化到 `~/.omlx/settings.json`，CLI 参数优先级高于配置文件。

---

架构概览

```

FastAPI Server (OpenAI / Anthropic API)

│

├── EnginePool（多模型、LRU 淘汰、TTL、手动加载/卸载）

│ ├── BatchedEngine（LLM，连续批处理）

│ ├── VLMEngine（视觉语言模型）

│ ├── EmbeddingEngine

│ └── RerankerEngine

│

├── ProcessMemoryEnforcer（总内存限制、TTL 检查）

│

├── Scheduler（FCFS，可配置并发）

│ └── mlx-lm BatchGenerator

│

└── Cache Stack

├── PagedCacheManager（GPU、块式、写时复制、前缀共享）

├── Hot Cache（内存层、回写）

└── PagedSSDCacheManager（SSD 冷层、safetensors 格式）

```
